# AnyApp 工作台前端重构计划

- 状态：`[partial]`（已归档）
- 创建日期：2026-07-22
- 归档日期：2026-07-23
- 项目：`AnyApp`
- 范围：前端视觉系统、页面结构、交互原语与桌面验收
- 执行边界：已完成代码迁移、下拉菜单、使用说明、自动门禁和 Tauri 技术 smoke；用户明确后续自行编辑视觉，因此以 `[partial]` 归档

## 背景

AnyApp 已完成 Tauri 2、React、TypeScript、JSON repository 和 Windows 本地能力，但当前前端属于功能优先的首版实现，存在以下问题：

- 全局 CSS 变量、`App.css` 硬编码颜色和组件内 Tailwind 色值同时存在，缺少单一视觉事实源。
- 软件条目采用大卡片并同时暴露过多操作，影响名称扫描和快速启动。
- 路径健康、筛选、分类、搜索、排序和数据状态在多个区域重复表达，主次关系不清晰。
- 弹窗、通知、设置卡片和表单组件由少量临时原语拼接，交互和视觉细节不统一。
- 现有小窗口策略主要依靠隐藏内容，未围绕 Tauri 最小窗口 `840 × 600` 重新组织信息。

用户已明确否定当前视觉结果，并确定采用一套由权利人授权的 source-first 工作台组件源作为新前端基础。该组件源与 AnyApp 的 React 19、TypeScript 7、Vite 8 和 Tailwind CSS 4 技术栈一致。

## 产品对象与单一任务

- **对象**：需要管理已安装软件和便携软件的 Windows 用户。
- **核心任务**：尽可能快地找到、判断路径状态并启动一个应用。
- **设计判断**：AnyApp 是桌面启动工具，不是数据大屏、展示 Gallery 或 SaaS 仪表盘。

所有布局、层级和动效都必须服务于“快速扫描与启动”，不能用指标卡、装饰性背景或多层容器抢占注意力。

## 已确认决策

1. 使用已获权利人授权的 source-first 组件源；不设置额外许可证审批门。
2. AnyApp 是潜在公开项目，计划、源码和文档不得写入私有组织名、私有仓库 URL、私有提交哈希或其他私有侧标识。
3. 采用按需源码安装，不整仓复制，不建立软链接、junction、子模块、共享源码或运行时同步关系。
4. 新 UI 行为原语统一到 Base UI；最终不得长期并存 Radix 与 Base UI 两套 Dialog、Switch、Tooltip 等基础行为。
5. 使用组件源的语义 token、`data-slot`、CVA、`cn()` 和工作台组件结构，但建立独立的 AnyApp 主题。
6. 不复制上游品牌锁定、品牌资产或 WebGL 抖动背景。
7. 保留 AnyApp 已确定的本地字体栈：
   - 标题：`Bahnschrift` / `Segoe UI Variable Display`。
   - 正文：`Segoe UI Variable Text` / `Segoe UI`。
   - 路径与数据：`Cascadia Mono` / `Cascadia Code`。
8. 本轮保持暗色工作台为默认体验；语义 token 不阻断未来亮色主题，但不主动增加主题切换功能。
9. 左侧路径健康轨是 AnyApp 的识别性元素，必须保留并改进，不能被通用组件风格抹掉。
10. Rust commands、JSON schema、TanStack Query 与 Zustand 职责边界保持不变；前端重构不得复制持久化状态。

## 视觉方向

### 色彩

以下十六进制仅用于表达目标色感；实施时转换为语义 OKLCH token，并以 token 为唯一事实源：

| 角色 | 参考值 | 用途 |
|---|---|---|
| `canvas` | `#0B111A` | 应用底色 |
| `surface` | `#121C2A` | 工具栏、行项目和弹层 |
| `surface-raised` | `#182538` | 选中、悬浮和重要弹层 |
| `primary` | `#3B82F6` | 启动、焦点和当前筛选 |
| `success` | `#34D399` | 路径正常 |
| `warning` | `#F59E0B` | 路径异常和需处理状态 |
| `foreground` | `#E6EDF7` | 主文本 |

禁止把颜色继续散落在组件 class 和独立 CSS selector 中。

### 布局

采用“紧凑导航轨 + 顶部全局搜索 + 高密度软件行”的工作台结构：

```text
┌ AnyApp ─────── 搜索软件 Ctrl+K ─────── 扫描  添加  设置 ┐
├────────┬───────────────────────────────────────────────┤
│ 全部   │ 全部 / 收藏 / 最近 / 路径异常                 │
│ 收藏   ├───────────────────────────────────────────────┤
│ 最近   │ ● 正常 18  ▲ 异常 2  分类：全部  排序：名称   │
│ 异常   ├───────────────────────────────────────────────┤
│ 分类   │▌ 图标  软件名称   分类 / 最近启动       ▶  ⋯ │
│        │        路径或一行描述                         │
│        ├───────────────────────────────────────────────┤
│        │▌ 图标  软件名称   分类 / 最近启动       ▶  ⋯ │
└────────┴───────────────────────────────────────────────┘
```

### 识别性元素

软件列表左侧使用连续、克制的路径健康轨：

- 正常：绿色细轨或状态点。
- 失效：琥珀色细轨。
- 不支持或越界：红色细轨。
- 键盘选中：钴蓝焦点框覆盖健康状态，但不能替代健康颜色。

这是本轮唯一允许强化的视觉特征；其他装饰保持安静。

### 动效

- 只保留工具栏状态切换、行项目悬浮/按压、弹层进入和通知反馈。
- 不增加持续环境动画、粒子、WebGL 或逐卡片延迟动画。
- `prefers-reduced-motion` 下移除位移和持续动画，功能和焦点顺序保持一致。

## 目标信息架构

### 一级结构

1. 顶部工具栏：品牌、全局搜索、扫描、添加、设置。
2. 左侧导航轨：全部、收藏、最近、路径异常、分类入口。
3. 内容工具栏：当前范围、分类、排序和紧凑状态汇总。
4. 软件列表：高密度行项目，优先显示名称、路径健康和启动入口。
5. 右侧设置 Sheet：启动行为、数据工具、备份、重新定位。

### 软件行操作优先级

- 始终显示：启动、收藏、更多菜单。
- 更多菜单：资源管理器定位、网站、桌面快捷方式、编辑、删除。
- 删除必须位于菜单末尾并与普通操作分隔。
- 路径异常时禁用启动，并把重新定位或编辑路径作为明确恢复动作。

## 范围

### 包含

- 授权组件源码的按需安装与本地化。
- 语义主题和基础行为原语统一。
- AppShell、搜索、筛选、软件列表和路径健康表达重构。
- Dialog、Sheet、AlertDialog、Menu、Toast、Field 和 Empty State 统一。
- `840 × 600` 与 `1200 × 800` 桌面布局。
- 键盘、焦点、reduced-motion、断网和 Tauri 桌面 smoke。

### 不包含

- Rust backend、JSON schema 或扫描算法重写。
- 新增在线商店、下载、账户、云同步或网络资源。
- 上游品牌素材、品牌锁定或 WebGL 背景。
- 安装包、自动更新、GitHub 发布、commit 或 push。
- 未经用户确认主动增加亮色主题。

## 总体验收标准

- 应用首屏一眼能识别搜索、软件列表和启动入口。
- 每个软件行最多暴露三个操作入口，不再展示七个平级按钮。
- 颜色、边框、圆角、阴影、字体和焦点全部来自语义 token 或统一组件变体。
- `App.tsx` 不再同时承担完整布局、筛选壳层、列表结构和所有弹层编排细节。
- 最终基础行为只保留 Base UI，不残留未使用的 Radix 依赖。
- `840 × 600` 下无横向滚动，搜索、筛选、启动和设置入口可达。
- 键盘 `Ctrl+K`、`/`、上下键和 Enter 启动行为保持可用。
- 用户文本仍只通过 JSX 渲染；外部链接仍在打开前显示完整目标。
- 不出现远程 CDN、私有侧标识、旧主题原型品牌或真实用户软件清单。
- 用户完成桌面视觉与交互验收后，才能把计划标为 `[done]`。

## 主要风险

| 风险 | 控制方式 |
|---|---|
| 一次性替换导致功能回归 | 先建立新组件层，再逐区域切换；旧组件在用户验收前不删除 |
| Base UI 与 Radix 混用 | 以功能区为单位迁移，最终执行依赖和 import 清零审计 |
| 上游主题覆盖 AnyApp 身份 | 仅复用语义结构，重新定义 AnyApp token、字体和路径健康轨 |
| 大量组件无差别导入 | 使用白名单和依赖闭包，只安装当前功能实际需要的 item |
| 私有侧信息进入公开仓库 | 计划、代码、注释、锁文件和报告执行专项去敏搜索 |
| 视觉自动化替代真实感受 | 自动门禁只验证功能与可访问性，最终视觉以用户人工确认为准 |

## 回滚原则

- 在新 AppShell 切换前保留现有 `App.tsx`、`App.css` 和临时 UI 组件。
- 新组件先进入独立目录并通过 build，再替换入口 import。
- 用户视觉验收前不删除旧壳层文件；发现问题时切回旧入口，不修改 Rust 或用户 JSON。
- 旧文件清理是最终独立步骤，必须在新界面通过自动门禁和用户确认后执行。

## 归档决定

- 工作台功能结构、Base UI 行为原语、软件行下拉菜单和内置使用说明均已落地。
- 用户明确认为当前视觉效果仍不理想，并决定后续自行编辑视觉；本计划不把该结果伪装为视觉验收通过。
- 计划以 `[partial]` 归档。后续视觉修改应建立新计划或由用户直接维护，不继续扩大本计划范围。
